SYSTEM MANIFEST

ARCHITECTURAL FOUNDATION: THE BARE METAL LAYER

K-DOS is an experimental, self-booting operating system designed for the X86_64 architecture. Unlike traditional OS builds that rely on heavy abstraction layers, K-DOS interacts directly with the CPU and memory controllers. The initialization sequence begins with a custom Stage 1 Bootloader written in NASM assembly. This 512-byte payload resides in the Master Boot Record (MBR) and is responsible for hardware detection and the transition from 16-bit Real Mode into 32-bit Protected Mode.

[SETUP_GDT]
gdt_start:
    dq 0x0000000000000000 ; Null descriptor
    dq 0x00af9a000000ffff ; Code descriptor
    dq 0x00cf92000000ffff ; Data descriptor
gdt_end:

MEMORY SEGMENTATION AND LONG MODE TRANSITION

The transition to 64-bit Long Mode is the most critical phase of the K-DOS kernel. This requires the manual setup of Page Tables (PML4, PDPT, PD, and PT). The kernel identity maps the first 2 megabytes of RAM to ensure the CPU can continue execution during the switch. Once the "Long Mode Enable" bit is flipped in the EFER model-specific register, K-DOS gains access to the full 64-bit instruction set and general-purpose registers (RAX, RBX, etc.).

Data persistence is managed through a custom FAT32 driver. This driver bypasses the standard BIOS interrupts in favor of direct LBA (Logical Block Addressing) via the ATA disk controller. By using I/O ports 0x1F0 through 0x1F7, K-DOS communicates directly with the storage hardware, achieving zero-latency data throughput. This is essential for the rapid loading of system binaries and user-space applications.

Interrupt handling is the backbone of the K-DOS user interface. The Interrupt Descriptor Table (IDT) contains 256 entries, each pointing to a specialized assembly stub. These stubs save the current CPU state (pushing all registers to the stack) before calling the high-level C logic. This ensures that even during heavy disk I/O, keyboard input remains responsive and system timing stays accurate to the millisecond.

HYBRID KERNEL PHILOSOPHY

KS-DOS 0.1 PRE-ALPHA is a community-centric distribution built upon a hardened Linux kernel backbone. The primary objective of KS-DOS is to eliminate the overhead associated with modern "systemd" based distributions. By utilizing a custom "init" system written in pure C, KS-DOS reaches a functional shell in under 2.5 seconds on NVMe-based hardware. This build utilizes the Multiboot2 protocol, allowing it to be loaded by a variety of modern bootloaders while maintaining its unique environmental variables.

# KS-DOS KERNEL PARAMETERS
BOOT_IMAGE=/boot/vmlinuz-ksdos
root=PARTUUID=01-alpha-ks
quiet splash loglevel=0 console=tty1

THE KS-SHELL AND USERLAND ENVIRONMENT

The user experience in KS-DOS is centered around the "KS-Shell," a lightweight command interpreter that prioritizes resource availability. Every system daemon is manually toggled; if a service is not actively requested by the user, it consumes zero CPU cycles and zero memory pages. This "On-Demand" architecture makes KS-DOS the ideal choice for developers who need maximum hardware performance for compilation and neural pathway testing.

The filesystem structure of KS-DOS follows a minimalist hierarchy. Unlike standard FHS (Filesystem Hierarchy Standard), KS-DOS consolidates binaries into a single high-speed path, reducing the search time for the execution linker. The package manager, currently titled "KS-GET," interacts directly with the git-based repositories to pull and compile source code on the fly, ensuring that every tool is optimized for the specific architecture of the machine it resides on.

Future development phases for KS-DOS 0.1 PRE-ALPHA include the integration of a custom Wayland compositor. This compositor, designed specifically for Tech Haven, will provide a seamless 2D/3D interface without the memory leaks inherent in legacy X11 systems. Security is also a top priority, with a built-in kernel-level sandbox that isolates networking processes from the core filesystem, effectively neutralizing unauthorized access attempts at the hardware interrupt level.
