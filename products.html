Tech Haven Studios — KOS™

KOS™ Kernel Engine

Technical Architecture

KOS is a bare-metal kernel written in x86_64 Assembly and C, following a microkernel-inspired design. The bootloader initializes hardware in under 3 seconds on modern systems using either UEFI or legacy BIOS. The kernel manages memory through a custom buddy-system page allocator, handles interrupts via the APIC, and abstracts hardware through a modular driver interface.

Current Features

  • 16/32/64-bit x86 support with UEFI and BIOS boot
  • 4KB/2MB/1GB page sizes with buddy allocator
  • AHCI/SATA storage drivers with DMA
  • PS/2 and USB keyboard and mouse input
  • VGA/VESA framebuffer graphics modes
  • Serial port debugging interface
  • PCI device enumeration
  • Experimental SMP multi-core scheduler

Roadmap

Q2 2026: Complete multi-core scheduling and SMP stability
Q3 2026: TCP/IP network stack implementation
Q4 2026: User-mode transition, system call interface, ELF loader

Tech Haven Studios — K-DOS™

K-DOS™ Operating System

Technical Architecture

K-DOS is a UNIX-like command-line OS built on top of the KOS kernel. It runs entirely from RAM with a footprint of approximately 8MB, making it ideal for embedded systems and testing environments. All core commands are built into the shell binary, avoiding filesystem dependencies during early boot. The shell implements a custom tokenizer, basic process scheduling, and a virtual filesystem layer.

Available Commands

  • ls, cd, mkdir, rmdir — filesystem navigation
  • cat, cp, mv — file read, copy and move
  • ps, kill — process listing and termination
  • help, clear, reboot, shutdown — system commands

Roadmap

Q2 2026: Shell scripting, pipes and I/O redirection
Q3 2026: Package manager for third-party utilities
Q4 2026: Network commands (ping, wget), persistent storage

Tech Haven Studios — K-DOS Desktop™

K-DOS Desktop™

Technical Architecture

K-DOS Desktop is an experimental GUI layer built on K-DOS using direct framebuffer rendering for maximum performance — no X11, no Wayland, just raw pixel writes. The stacking window manager handles input through a kernel-level event system that translates keyboard and mouse interrupts into GUI interactions. Double buffering prevents screen tearing during window redraws.

Current Progress

  • Basic window rendering with title bars ✓
  • Mouse cursor with custom themes ✓
  • Keyboard input in GUI mode ✓

Tech Haven Studios — BOOTOS™

BOOTOS™

What is BOOTOS?

BOOTOS is a brand new operating system built using the Multiboot2 specification and the GRUB bootloader. Unlike KOS which uses a custom bootloader written entirely in Assembly, BOOTOS leverages GRUB's mature, battle-tested boot infrastructure for maximum hardware compatibility from day one.

Why Multiboot2 and GRUB?

  • GRUB supports virtually all x86 hardware ever made
  • Multiboot2 gives the kernel a standardized interface for receiving memory maps, framebuffer info and hardware data
  • No need to maintain a custom bootloader — focus goes directly into the OS
  • Easier to test on real physical hardware from the start
  • Built-in support for GRUB multi-OS boot menus

Technical Architecture

BOOTOS is written in C with x86_64 Assembly for low-level hardware interaction. The Multiboot2 header is embedded in the kernel binary. When GRUB loads BOOTOS, it passes a structured information table to the kernel entry point containing the full memory map, framebuffer configuration, boot modules, and command line arguments.

Current Status

BOOTOS is 0-10% complete. The Multiboot2 kernel entry point and basic VGA text output are the current development focus. No downloadable build is available yet. Follow the Discord and blog for updates as development progresses.

Roadmap

  • Phase 1: Multiboot2 entry + VGA text output
  • Phase 2: Memory management + interrupt handling
  • Phase 3: Basic shell and filesystem
  • Phase 4: GRUB menu for multi-OS boot
